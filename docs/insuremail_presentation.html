Overall Solution Overview

End-to-End Flow — from Gmail Inbox to Routed Response

Gmail Inbox → IMAP Poller → S3 .eml → Step Functions → 6 steps → SES / Human Review

3-Tier Confidence Routing

≥ 0.80 → Auto-Response
Sent automatically via Amazon SES. Zero human involvement.
0.50 – 0.80 → Human Review
Queued for agent review with AI draft pre-filled.
< 0.50 → Escalate
Forwarded to specialist team with partial context.

Graceful Fallbacks

  • FallbackBioBERT — if SageMaker unavailable, LLM result used downstream
  • SetEmptyRAG — if retrieval times out, continue with empty context
  • SetEmptyCRM — if customer lookup fails, pipeline continues
  • Pipeline never hard-fails mid-run — worst case is escalation with partial data

Infrastructure as Code

  • Every Lambda, DynamoDB table, API Gateway route defined in Terraform
  • Reproducible, auditable, deploy with one command

Gmail Poller Scheduled Task

At-Least-Once Email Delivery · Fully Autonomous

How It Works

  • EventBridge scheduled rule fires every 5 minutes
  • Opens IMAP SSL to Gmail using App Password auth
  • SEARCH UNSEEN — only new, unprocessed emails
  • Downloads raw RFC 2822 .eml (full headers + body)
  • Uploads to S3 key incoming/gmail-{uuid}.eml
  • Fires Step Functions execution with S3 key in payload

Reliability Guarantees

At-Least-Once Delivery
Email only marked \Seen after successful S3 upload + trigger. Failure = stays unread = retried next poll.
No Manual Intervention
Zero human steps to get emails into the pipeline. Self-waking and autonomous.
Future: Gmail Push API
Replace IMAP polling with real-time push notifications — <1 second latency.

Gmail Processing — Full Step Functions Pipeline

6 Lambda steps · Parallel dual-classifier · Graceful fallbacks at every stage

STEP 1
email_parser Lambda
RFC 2822 MIME parse · PII redact · PDF→Textract OCR · DOCX/TXT extract · Mistral 7B inline entity extraction (policy_number, claim_amount, membership_no, 30+ Laya form fields)
STEP 2 — PARALLEL
classify_intent_by_llm ‖ classify_intent_by_biobert
LLM: Mistral 7B → 17 intents → 12 teams · BioBERT: SageMaker serverless endpoint, hardcoded alphabetical labels · FallbackBioBERT on timeout
STEP 3
rag_retrieval Lambda
HyDE: Mistral 7B generates hypothetical answer → embed · BM25 keyword + Titan V2 vector (1024-dim) · RRF fusion · Mistral 7B cross-encoder rerank → top 3–5 policy doc chunks
STEP 4
crm_validation Lambda
Mistral 7B Text-to-SQL → DynamoDB customers/policy lookup · Validates: customer exists, policy active, coverage in scope · SetEmptyCRM fallback on error
STEP 5
llm_response Lambda
Mistral 7B / Llama 3.1 8B (fallback) generates personalised reply grounded in RAG docs + CRM context · 8-dim quality judge scores output → final confidence_score
STEP 6
email_sender SES
≥0.8 → auto via Amazon SES · 0.5–0.8 → human review queue · <0.5 → escalate · Result → DynamoDB email_processing → Dashboard

Knowledge Base Ingestion

The Memory of InsureMail AI — Grounding Every Response in Real Policy Documents

Data Source — Laya Healthcare Crawler

  • scripts/crawl_laya.py — targets public Laya Healthcare website
  • Extracts: policy terms, coverage guides, claim procedures, pre-auth requirements, FAQs, network provider info
  • Output: structured .jsonl knowledge documents
  • Loaded via scripts/load_knowledge_docs.py

Why This Matters

Without a populated KB, the response generator produces plausible but generic answers. With it, every reply is grounded in the actual policy documents that apply to that customer's specific query.

Ingestion Pipeline

500-Token Chunking, 50-Token Overlap
No sentence cut at boundaries. Semantic continuity preserved.
MD5 Hash Deduplication
Never re-embeds identical content. Idempotent — safe to re-run on website updates.
Titan Embeddings V2 — 1024-dim
14 document categories stored in DynamoDB kb_embeddings.

BioBERT Fine-tuning for Insurance Intent

Base: dmis-lab/biobert-base-cased-v1.2 — Pre-trained on PubMed + PMC

Why BioBERT?

  • Insurance language contains medical terminology — ICD codes, drug names, clinical phrases like "pre-authorisation"
  • BioBERT pre-trained on biomedical literature gives a head start vs general BERT
  • Fine-tuned on 17-class insurance intent taxonomy — aligns exactly with LLM classifier set

Key Engineering Lesson

Label Order = Hardcoded Alphabetical
BIOBERT_LABELS list fixed in alphabetical order in Lambda. If order changes between training and inference, every prediction is wrong. Version-safe by design.

17 Intent Classes (Alphabetical)

broker_query cancellation_request claim_reimburse_query claim_status claim_submission complaint coverage_query dependent_addition document_followup enrollment_new_policy hospital_network_query id_verification other payment_issue policy_change pre_authorisation renewal_query
100% accuracy — 20-email eval Runs parallel with LLM on every email
